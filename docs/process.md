# 프로세스

이 문서는 캠프를 어떻게 만들고 운영하는지 설명한다. 새 캠프를 만들 때 이 문서를 따라가면 된다.

---

## 1. 왜 프로세스 문서가 필요해?

캠프를 만들 때마다 "이거 어떻게 했더라?" 하면 비효율적이다. 프로세스가 정해져 있으면:
- 매번 처음부터 고민 안 해도 된다
- 빠뜨리는 게 없다
- 다른 사람(또는 Claude)도 따라할 수 있다

---

## 2. 캠프 제작 전체 흐름

```
1. 기획 → 2. 제작 → 3. 테스트 → 4. 모집 → 5. 운영 → 6. 회고
```

각 단계를 자세히 설명한다.

---

## 3. 기획 단계

### 뭘 정해야 해?

새 캠프를 시작하기 전에 이것들을 먼저 정해야 한다:

| 항목 | 질문 | 예시 (AI 포트폴리오) |
|------|------|---------------------|
| 주제 | 뭘 만드는 캠프야? | 포트폴리오 웹사이트 |
| 타겟 | 누구를 위한 거야? | 비전공자, 취준생, 이직러 |
| 기간 | 며칠이야? | 6일 |
| 결과물 | 참가자가 뭘 가져가? | 배포된 포트폴리오 URL |
| 콜라보 | 누구와 함께해? | 크크웍스 |
| 목표 | 뭘 검증하려고 해? | AI로 비전공자도 가능한지 |

### 레슨 플랜

기획이 끝나면 **레슨 플랜**을 작성한다. 레슨 플랜은 각 Day의 설계도다.

위치: `documents/lesson-plans/day-{n}.md`

레슨 플랜에 들어가는 것:
- Day의 목표
- 핵심 메시지
- 스텝 구성
- 미션 설계
- 예상 소요 시간

**중요**: 콘텐츠 작성 전에 레슨 플랜을 먼저 완성해야 한다. 레슨 플랜 없이 바로 쓰면 방향이 흔들린다.

## 4. 제작 단계

### 폴더 구조

새 캠프를 만들면 이런 구조가 생긴다:

```
src/app/cc/[캠프명]/
├── page.tsx              # 랜딩 페이지
├── apply/
│   └── page.tsx          # 신청 페이지 (챗봇)
└── workbook/
    ├── page.tsx          # 워크북 메인
    └── day/[day]/
        ├── page.tsx      # Day 페이지
        └── _content/
            ├── day1.ts   # Day 1 콘텐츠
            ├── day2.ts
            └── ...
```

### 콘텐츠 작성

Day 콘텐츠는 TypeScript 파일로 작성한다:

```typescript
// day1.ts
export const day1: DayContent = {
  title: 'Day 제목',
  subtitle: '부제목',
  steps: [
    {
      id: 'step-1',
      title: '스텝 제목',
      duration: '5분',
      content: `마크다운 내용`
    }
  ],
  mission: {
    title: '미션 제목',
    description: '미션 설명',
    fields: [...]
  }
}
```

자세한 콘텐츠 작성법은 `CLAUDE.md`의 "워크북 콘텐츠 작성 규칙" 참고.

### 에이전트 활용

콘텐츠 제작할 때 Claude 에이전트를 활용할 수 있다:

```
/research day{n} → /write day{n} → /edit day{n} → /test day{n}
```

| 에이전트 | 역할 | 산출물 |
|----------|------|--------|
| 리서쳐 | 자료 수집, 도구 조사 | research-day{n}.md |
| 워크북 제작자 | Day 콘텐츠 작성 | day{n}.ts |
| 편집자 | 톤앤매너, 흐름 검토 | day{n}.ts 수정 |
| 유저 테스터 | UX 검증, 문제 발견 | test-day{n}.md |

### 랜딩 페이지

캠프마다 전용 랜딩 페이지가 필요하다. 랜딩에 들어가는 것:
- 캠프 소개 (뭘 하는 건지)
- 일정 (언제, 며칠)
- 결과물 (뭘 가져가는지)
- 신청 버튼

## 5. 테스트 단계

론칭 전에 확인해야 할 것들:

### 콘텐츠 체크
- [ ] 모든 Day 콘텐츠 완성
- [ ] 오타/문법 확인
- [ ] 이미지/링크 작동 확인
- [ ] 모바일에서 잘 보이는지

### 기능 체크
- [ ] 신청 플로우 테스트
- [ ] 미션 제출 테스트
- [ ] Slack 알림 오는지
- [ ] Firebase 데이터 저장되는지

### 배포 체크
- [ ] Vercel 환경변수 설정
- [ ] 도메인/URL 확인
- [ ] OG 이미지 (SNS 공유용)

## 6. 모집 단계

### 신청 받기

신청은 챗봇 형태로 받는다 (`/apply` 페이지). 챗봇이 기본 정보를 물어보고, 제출하면:
1. Firebase에 저장
2. Slack에 알림

### 홍보

홍보 채널 예시:
- 콜라보 파트너의 채널 (핵심)
- 실험마켓 SNS
- 관련 커뮤니티
- 유료 광고 (선택)

### 참가자 안내

신청자에게 시작 전 안내:
- 시작 일시
- 워크북 접속 방법
- 준비물 (있으면)

---

## 7. 운영 단계

### Day별 오픈

캠프가 시작되면 매일 새 Day를 오픈한다.

**오픈 방법**:
`workbook/page.tsx`에서 `isLocked` 조건 수정 → 배포

예: `mission.day > 3` → Day 4부터 잠금

### 미션 확인

참가자가 미션을 제출하면 Slack에 알림이 온다. 확인하고 필요하면 피드백.

### Q&A 대응

참가자 질문이 오면 답변. 자주 묻는 질문은 콘텐츠에 반영.

---

## 8. 회고 단계

캠프가 끝나면 회고를 한다.

### 데이터 정리

| 항목 | 내용 |
|------|------|
| 신청자 수 | - |
| 완주자 수 | - |
| 완주율 | - |
| Day별 이탈 | - |

### 회고 질문

- 뭐가 잘 됐어?
- 뭐가 아쉬웠어?
- 참가자 피드백은?
- 다음에 뭘 바꿔야 해?

### 문서화

회고 결과를 [캠프 현황](camps.md)에 기록.

---

## 9. 자주 하는 실수

캠프 운영하면서 자주 하는 실수들:

| 실수 | 해결 |
|------|------|
| 레슨 플랜 없이 바로 쓰기 | 레슨 플랜 먼저 완성 |
| Day 분량이 너무 많음 | 30분 이내로 끝나게 |
| 미션이 너무 어려움 | 작게 쪼개기 |
| 테스트 안 하고 론칭 | 체크리스트 활용 |
| 데이터 안 모음 | 처음부터 지표 정의 |

```markdown
-

-
```
