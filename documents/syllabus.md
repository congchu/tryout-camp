# AI 포트폴리오 5일 챌린지 - 실라버스

## 챌린지 목적

### 대외 메시지
> **AI로 5일 만에 나만의 웹 포트폴리오 만들기**

프리랜서에게 포트폴리오는 필수다. 하지만 현실은:
- 노션이나 PDF로 대충 정리해둔 상태
- "나중에 제대로 만들어야지" 하면서 미룸
- 웹 포트폴리오는 코딩/디자인 몰라서 포기

AI 도구가 발전하면서 **코딩/디자인 몰라도 웹사이트를 만들 수 있는 시대**가 됐다.
이 챌린지에서 직접 만들어본다.

---

### 내부 의도 (기획용 - 커리큘럼에 드러나지 않음)

**진짜 목적: AI와 친해지기의 시작점**

포트폴리오는 "결과물"로 선택한 것.
- 프리랜서라면 어차피 필요한 것
- 완성하면 실제로 쓸 수 있음
- 의미 있는 결과물이 있어야 동기부여가 됨

**현실적 한계**:
- 5일 안에 포트폴리오 "완성"은 어려움
- 이 챌린지는 **"시작"**에 목적이 있음

**궁극적 비전**:
1. 이 챌린지로 AI와 친해지기 (입문)
2. 계속 포트폴리오/이력 업데이트하면서 AI 활용 능력 ↑
3. 진짜 홍보용으로 각잡고 쓸 수 있는 수준까지

**후속 과정 필요**:
- 5일은 입문 (아쉬움이 남을 것)
- 심화/실전 과정으로 이어져야 함

---

## 이 챌린지의 목표

### 대외 목표
> 5일 후, 클라이언트에게 **링크 하나로** 나를 보여줄 수 있는 상태

- 실제 URL로 접속 가능한 웹 포트폴리오
- 내 정보와 작업물이 담긴 상태
- 필요할 때 직접 수정 가능

### 내부 목표 (기획용)
> **"AI로 이런 것도 할 수 있구나"** 를 직접 경험하기

- AI 도구와 친해지는 첫 경험
- "이게 된다고?" 하는 순간들
- "나도 이런 거 할 수 있네" 자신감
- → 후속 과정으로 이어지는 시작점

---

## 왜 5일인가

### 5일의 의미
- 너무 짧으면: 대충 만들고 끝 → 실제로 안 씀
- 너무 길면: 중간에 포기 → 완성 못 함

5일은 **"진짜 쓸 수 있는 수준"까지 가는 최소 시간**.

### 하루 30~40분
- 직장인/프리랜서가 저녁에 할 수 있는 분량
- 너무 짧아서 의미없는 것도 아니고
- 너무 길어서 부담스럽지도 않은 선

---

## 5일 구조와 각 Day의 의도

### 전체 흐름
```
Day 1-2: 도구 익히기 (멋진 껍데기 만들기)
Day 3: 본질 작업 (진짜 내 포트폴리오로)
Day 4-5: 마무리 (세상에 공개 + 확인)
```

---

### Day 1: AI로 웹사이트 만드는 법

**의도**: "AI로 이게 된다"는 경험을 먼저 시키자

왜 첫날에 바로 만들어보게 하는가?
- 설명만 듣다 보면 지루해짐
- "일단 해봐야" 감이 잡힘
- 결과물이 나와야 동기부여가 됨

왜 "왜 포트폴리오가 필요한지"부터 시작하는가?
- 목적 없이 만들면 방향을 잃음
- "멋있어서"가 아니라 "필요해서" 만드는 거라는 인식

**핵심 학습**:
- 프롬프팅 = AI한테 잘 설명하기
- 레퍼런스가 있으면 결과가 좋아짐

**한계 인식**:
- AI 빌더로는 "비슷하게"는 되지만 "똑같이"는 어려움
- 이 한계를 Day 2에서 해결

---

### Day 2: 화려한 포폴 따라만들기

**의도**: Day 1의 한계를 넘어서, "진짜 멋진 사이트"를 만들 수 있다는 걸 보여주자

왜 GitHub/Codex를 가르치는가?
- 스크린샷으로 설명하는 건 한계가 있음
- 설계도(코드)를 직접 가져오면 "똑같이" 가능
- 개발자가 아니어도 Codex가 다 해줌

왜 "복사"부터 시작하는가?
- 백지에서 시작하면 막막함
- 좋은 레퍼런스를 복사한 후 수정하는 게 더 쉬움
- "모방 → 변형 → 창작" 순서

**이 시점의 상태**:
- 멋진 웹사이트가 내 컴퓨터에서 돌아감
- 하지만 아직 **남의 정보**가 들어있음
- → Day 3에서 진짜 내 것으로

---

### Day 3: 콘텐츠 완성하기

**의도**: 이제 "껍데기"를 "진짜 내 포트폴리오"로 만들자

왜 이 시점에서 "목적 정하기"를 하는가?
- Day 1-2에서 도구에 익숙해진 상태
- 이제 "뭘 넣을지" 고민할 차례
- 멋지게만 만들면 안 되고, **목적에 맞아야** 함

**"나 위키"란?**

내 모든 자료를 한 곳에 쏟아붓는 것.
- 이력서, 프로젝트 자료, 작업물, 메모 등
- 정리 안 돼도 괜찮음. 일단 다 넣기.

**왜 이 과정이 필요한가?**

1. **백지 공포 해결**
   - "포트폴리오에 뭘 써야 하지?" → 막막함
   - 자료를 먼저 쏟아내면 **시작점**이 생김
   - 정리는 나중에, 일단 재료부터

2. **AI한테 맥락 주기**
   - AI는 내가 뭘 해왔는지 모름
   - 자료를 주면 → 맥락을 이해하고 → 더 좋은 결과
   - "나에 대해 아는 AI"가 됨

3. **정리의 부수 효과**
   - 쏟아내다 보면 "내가 이런 것도 했네?" 발견
   - 흩어져 있던 것들이 한 곳에 모임
   - 나중에 다른 용도로도 활용 가능

**왜 "위키"라는 이름인가?**
- 위키피디아처럼 모든 정보가 한 곳에
- 서로 연결되어 있고
- 언제든 추가/수정 가능
- 비개발자도 이해하는 단어

왜 인터뷰 형식인가?
- 백지에서 글 쓰기는 어려움
- 질문에 답하다 보면 자연스럽게 정리됨
- AI가 "클라이언트 관점"으로 다듬어줌

**이 시점의 상태**:
- 내 정보가 들어간 진짜 포트폴리오
- 하지만 아직 **내 컴퓨터에서만** 보임
- → Day 4에서 세상에 공개

---

### Day 4: 배포하기

**의도**: 내 컴퓨터에서만 보이던 걸 **세상에 공개**하자

왜 Netlify인가?
- 무료
- **GitHub 없이** 드래그 앤 드롭으로 배포 가능
- 커스텀 도메인 가능

배포의 의미:
- "localhost"가 아닌 실제 URL
- 누구에게나 링크 공유 가능
- 이제 진짜 "포트폴리오"가 됨

**이 시점의 상태**:
- 실제 URL로 접속 가능
- 하지만 **누가 보는지 모름**
- → Day 5에서 확인

---

### Day 5: 방문자 확인

**의도**: 만들고 끝이 아니라, **반응을 확인**할 수 있게 하자

왜 분석을 넣는가?
- 아무도 안 보면 의미 없음
- 몇 명이 봤는지, 어디서 이탈하는지 알아야 개선 가능
- "데이터 기반 개선" 습관

왜 Mixpanel인가?
- 무료 티어 충분
- 이벤트 기반이라 유연함
- 설치 간단

**챌린지 완료 상태**:
- 실제 URL이 있고
- 분석이 연동되어 있고
- 필요할 때 직접 수정 가능
